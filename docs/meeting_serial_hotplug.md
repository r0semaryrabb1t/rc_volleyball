# 专家会议纪要：串口热插拔问题

## 问题描述
不关闭 `motor_control_node` 的情况下，断电后重新给电机上电，只有 1 台电机能正常反馈数据（3 台中只有 1 台在线）。

## 现象分析
```
arm_motor_1  GO-8010  ID=0  离线  30Hz
arm_motor_2  GO-8010  ID=0  在线  30Hz
arm_motor_3  GO-8010  ID=0  离线  30Hz
```
3 台电机通过独立串口通信，但热插拔后只有 1 台恢复在线。

---

## Sub-Agent A（嵌入式通信工程师）

> **根因定位：内核 RX 缓冲区残留数据**
>
> `sendRecvAccumulate()` 的流程是：发送 17 字节命令 → 等待 10ms → 累积接收 16 字节反馈。
>
> 问题在于：电机断电期间，`send()` 正常发出，`receive()` 超时返回 0。但电机**重新上电的瞬间**，可能回传不完整的字节（上电自检/随机噪声），这些字节残留在 Linux 内核的串口 RX FIFO 中。
>
> 下一次 `sendRecvAccumulate()` 调用时：
> 1. 先发送新命令（17 字节）
> 2. `receive()` 先读到了**旧的残留字节**（可能是 1-5 个随机字节）
> 3. 紧接着读到正确的反馈帧，但由于前面的残留字节，整个 16 字节窗口**偏移**了
> 4. `parseFeedback()` 检查 `data[0] != 0xFD || data[1] != 0xEE` → **帧头不匹配** → 返回 false
> 5. 电机被标记为离线
>
> 只有 1 台碰巧缓冲区干净（残留字节恰好为 0 或刚好被前一次读取消耗），所以仅 1 台在线。
>
> **修复方案：在每次 send 前调用 `tcflush(fd_, TCIFLUSH)` 清空内核 RX 缓冲区。**
> 这是 POSIX 标准做法，开销极低（一个 ioctl），确保每次事务从干净状态开始。

## Sub-Agent B（测试/运维工程师）

> **质疑：tcflush 够用吗？会不会丢掉正在传输的有效数据？**
>
> A 的方案有个边界条件：如果控制频率很高（200Hz），而电机反馈恰好在 tcflush 的瞬间到达，会不会丢掉有效帧？
>
> 分析：
> - 宇树电机 RS485 是**严格的请求-响应协议**（半双工）
> - 电机**只有在收到命令后才会回传**反馈
> - `tcflush` 在 `send()` 之前执行，此时不可能有"正在到达的有效反馈"
> - 唯一可能残留的数据就是：上一轮超时未读完的尾部字节、或上电噪声
>
> **结论：tcflush 在请求-响应协议下是安全的。** 但建议增加一个诊断日志：当 tcflush 前检测到缓冲区有数据时，记录丢弃字节数（方便现场调试）。
>
> 不过，诊断日志的优先级低，不阻塞修复。

## Sub-Agent C（系统架构师）

> **折中建议：先修 tcflush，后续可选帧同步**
>
> 同意 A 的根因分析和 B 的安全性论证。tcflush 是最小侵入性修复。
>
> 更鲁棒的方案是在 `parseFeedback()` 或累积接收逻辑中增加**帧同步**（在接收到的数据中搜索 0xFD 0xEE 帧头并对齐），但这会增加复杂度，且在 tcflush 已经解决根因的情况下，收益有限。
>
> **建议分两步走：**
> 1. **立即修复**：`tcflush(fd_, TCIFLUSH)` 加在 `sendRecvAccumulate()` 和 `sendRecv()` 的 send 前
> 2. **后续增强**（可选）：如果实测中仍偶发帧错位，再加帧同步逻辑

---

## 最终共识

三位专家**一致赞同**方案：

| 项目 | 内容 |
|------|------|
| **修复方式** | 在 `sendRecvAccumulate()` 和 `sendRecv()` 的 `send()` 调用前加 `tcflush(fd_, TCIFLUSH)` |
| **修复位置** | `src/motor_control_ros2/src/hardware/serial_interface.cpp` |
| **安全性** | RS485 请求-响应协议下 tcflush 不会丢有效数据 |
| **风险** | 极低（标准 POSIX 调用，开销可忽略） |
| **验证方法** | 不关闭节点，反复断电/上电电机，观察 3 台是否均恢复在线 |

## 修改已完成
- `sendRecvAccumulate()`：在 send 前增加 `tcflush(fd_, TCIFLUSH)`
- `sendRecv()`：同上
