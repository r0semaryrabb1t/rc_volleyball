# Delta 机械臂"高度过高"问题分析

## 现象
监控器显示：当前高度 **0.430m**，目标高度 0.250m，电机全部离线。

## 数据溯源

### 日志数据
```
配置: a=0.260 b=0.250 c=0.370 r=0.380
电机位置(rad): arm_motor_1=2.778, arm_motor_2=2.147, arm_motor_3=3.034
电机offset(rad): 0.639, 0.063, 0.872
direction=1
```

### 反向变换验证（controlLoop 第 286 行）
```
theoretical_rad = fb.position × direction − offset
```
| 电机 | position | offset | 运动学角(rad) | 运动学角(°) |
|------|----------|--------|-------------|-----------|
| arm_motor_1 | 2.778 | 0.639 | 2.139 | 122.5° |
| arm_motor_2 | 2.147 | 0.063 | 2.084 | 119.4° |
| arm_motor_3 | 3.034 | 0.872 | 2.162 | 123.9° |
| **平均** | | | **2.128** | **121.9°** |

### 正运动学验证（FK: θ → h）
```
θ = 121.9°
cos(α) = (b·cos(θ) + c − r) / a
       = (0.250×(−0.528) + 0.370 − 0.380) / 0.260
       = −0.142 / 0.260 = −0.546
α = 123.1°, sin(α) = 0.838
h = b·sin(θ) + a·sin(α)
  = 0.250×0.849 + 0.260×0.838
  = 0.212 + 0.218 = 0.430m ✓
```
**结论：0.430m 的计算在数学上是正确的。**

### 目标对比
目标 h=0.250m → θ=150.3°，期望的电机位置：
| 电机 | 期望(rad) | 实际(rad) | 差值(rad) | 差值(°) |
|------|-----------|-----------|-----------|---------|
| arm_motor_1 | 3.263 | 2.778 | −0.485 | −27.8° |
| arm_motor_2 | 2.687 | 2.147 | −0.540 | −30.9° |
| arm_motor_3 | 3.496 | 3.034 | −0.462 | −26.5° |

三个电机均偏离目标约 **0.5 rad (28°)**。

---

## 根因

### Bug: controlLoop 中的陈旧数据问题

`delta_arm_node.cpp` controlLoop() 中：

```cpp
// 修复前
if (angle_count > 0) {
    current_joint_angle_ = ...;
    current_height_ = ...;
}
// 当 angle_count == 0（所有电机离线）时，不进入此块
// current_height_ 保持最后一次在线时的值 → 陈旧数据
```

**时间线：**
1. 电机通电 → `online=true`，计算得 current_height = 0.430m
2. 电机关闭 → `online=false`
3. controlLoop 中 `fb.online` 检查导致 `angle_count=0`
4. `current_height_` 被冻结为 0.430m，永不更新
5. 发布到 `/delta_arm_state` → 监控器显示陈旧值

### 修复
```cpp
// 修复后
if (angle_count > 0) {
    current_joint_angle_ = ...;
    current_height_ = ...;
} else {
    // 所有电机离线时，清零当前值
    current_joint_angle_ = 0.0;
    current_height_ = 0.0;
}
```

### 附加观察：arm_motor_2 异常
- `error=5`, `velocity=−1.50 r/s`（离线状态下不应有非零速度）
- offset 仅 0.063 rad，远小于另两个电机（0.639, 0.872），可能安装补偿校准有误
- 建议重新校准 arm_motor_2 的 offset

---

## 专家研讨

### Sub-Agent A（控制工程师）
> 离线时清零是最安全的做法。0.430m 是陈旧值——电机在通电时的物理位置决定了这个数值。电机没有到达目标说明：(a) 命令发出时电机已离线，或 (b) kp=0.20 不够对抗重力负载。建议增加重力补偿力矩前馈。

### Sub-Agent B（嵌入式/运维工程师）
> 清零不够，应该用 NaN 或特殊值标记"数据不可用"，清零容易与"真的在零位"混淆。而且 arm_motor_2 的 error=5 需要排查：Unitree GO-M8010 error code 5 通常是过压/通信超时。offset=0.063 也可疑。

### Sub-Agent C（架构师）
> 折中方案：先清零修复紧急问题（monitor 的 `online=false` 已区分了"零位"和"离线"），后续再考虑 NaN。error=5 是独立问题，不阻塞本次修复。offset 校准需要物理测量，建议用户实地验证。

### 结论
三人一致赞同：
1. **先清零**修复陈旧数据 bug（已完成）
2. arm_motor_2 的 error=5 和 offset 校准作为后续排查项

# 问题二：位置超调与 offset 异常 — 深度分析

## 核心问题

电机实际位置偏离目标约 **0.5 rad (28°)**，始终无法到达目标高度 0.250m。

## 根因分析

### 根因 1: kp 过小，位置环力矩不足以抗重力

当前 `delta_arm.yaml` 中 `kp: 0.02`。

**GO-M8010-6 电机端 FOC 力矩公式**：
```
τ = kp × (pos_target - pos_actual) + kd × (vel_target - vel_actual) + torque_ff
```

**定点数传输公式**（`unitree_motor_native.cpp` 第 102 行）：
```cpp
k_pos_q15 = kp × 1280.0    // kp=0.02 → k_pos_q15 = 25.6 → 量化为 25
```

**实际力矩计算**（电机侧还原）：
```
电机端位置误差 = 0.5 rad × 6.33(减速比) = 3.165 rad(电机轴)
pos_error_q15 = 3.165 × 32768 / (2π) ≈ 16517

τ_from_kp = k_pos_q15 × pos_error_q15 / (32768 × 256)
          = 25 × 16517 / 8388608
          ≈ 0.049 Nm
```

**对比**：GO-M8010-6 额定力矩 23.5 Nm（输出轴），**0.049 Nm 仅为额定的 0.2%**，完全不足以驱动机械臂抗重力。

### 根因 2: arm_motor_2 的 offset 异常

| 电机 | offset (rad) | offset (°) | 备注 |
|------|-------------|-----------|------|
| arm_motor_1 | 0.639 | 36.6° | 合理范围 |
| arm_motor_2 | **0.063** | **3.6°** | ⚠️ 异常偏小 |
| arm_motor_3 | 0.872 | 50.0° | 合理范围 |

三台电机同型号、同结构安装，offset 应在同一量级。arm_motor_2 的 0.063 与其他两台差距 **10 倍以上**，有两种可能：

**假设 A**：校准时 arm_motor_2 恰好安装在接近零位 → 真实 offset 就是 0.063
- 概率：低。三台同结构安装，零位偏差应相近

**假设 B**：校准过程有误（如取了错误的参考角度、或校准时电机处于 error 状态）
- 概率：高。error=5 说明该电机通信异常，校准时的反馈值可能不准确

### 根因 3: error=5 的含义

GO-M8010-6 error code 位于反馈帧 byte[12] 低 3 位（`MError`）：

| 错误码 | 含义 |
|--------|------|
| 0 | 正常 |
| 1 | 过温 |
| 2 | 过流 |
| 3 | 过压 |
| 4 | 编码器异常 |
| **5** | **已废弃/保留** 或 **通信超时**（固件版本依赖） |

error=5 最可能是**通信超时**——电机未在规定时间内收到下一帧命令，自动进入保护。这与问题一（串口热插拔）直接关联。

## 专家研讨会议

### Sub-Agent A（控制工程师）

> **kp=0.02 绝对是首要问题。**
>
> 我做了定量计算：要把 Delta 臂（假设负载质量 1kg，力臂 0.25m）悬停在目标位置：
> - 需要抗重力力矩 ≈ 1.0 × 9.81 × 0.25 ≈ 2.45 Nm
> - 当前 kp=0.02 在 0.5rad 误差下只产生 ~0.05 Nm
> - **差了约 50 倍！**
>
> 建议方案：
> 1. kp 提高到 **0.5~1.0** 范围（试验从 0.5 开始）
> 2. kd 保持 0.08 或适当提高到 0.10（阻尼防振荡）
> 3. 增加**重力补偿前馈**：`torque_ff = m × g × L × cos(θ)` ≈ 固定 0.3 Nm
>
> 这样位置环就能真正把臂压到目标位置。

### Sub-Agent B（嵌入式/运维工程师）

> **等一下，kp 从 0.02 直接跳到 0.5 太危险了！**
>
> 如果 offset 本身就是错的（arm_motor_2 差了 30°），高 kp 会让电机以大力矩冲向错误位置，可能：
> 1. 电机堵转过流 → 烧电机
> 2. 机械臂结构冲击 → 损坏传动机构
> 3. 三台电机目标不一致（因 offset 不同）→ 结构内力撕扯
>
> **必须先修正 offset，再提 kp！** 顺序不能反。
>
> 我主张：
> 1. 先解决问题一（串口热插拔），确保 3 台电机都在线
> 2. 用低 kp（当前 0.02）手动校准 arm_motor_2 的 offset
> 3. 确认 3 台 offset 合理后，再逐步提 kp（0.02 → 0.1 → 0.3 → 0.5）

### Sub-Agent C（架构师）

> 两位说得都对。我来梳理优先级：
>
> **B 的安全顾虑是正确的**——高 kp + 错误 offset = 灾难。A 的物理分析也是对的——kp=0.02 根本不可能到位。
>
> 折中方案：
>
> **阶段 1（安全恢复）**：
> 1. 修复串口热插拔（问题一），确保 3 台电机稳定在线
> 2. 用当前低 kp=0.02（安全力矩），逐台手动验证 offset：
>    - 给每台电机发送 θ=0° 命令，观察物理角度是否一致
>    - 如果 arm_motor_2 位置明显偏离另两台 → 重新校准 offset
>
> **阶段 2（性能优化）**：
> 3. offset 校准正确后，kp 从 0.02 → 0.10（5倍提升）
> 4. 观察有无振荡，如果稳定 → 0.10 → 0.30 → 0.50
> 5. 同步增加 kd（kd ≈ kp × 2~4 是经验值），如 kp=0.30 时 kd=0.60~1.20
>
> **阶段 3（可选增强）**：
> 6. 增加重力补偿前馈 `torque_ff`，进一步减少稳态误差
> 7. 考虑添加积分项（但嵌入式 FOC 通常不推荐 ki）

### 讨论

**A**：我同意先修 offset 再提 kp。补充：阶段 2 的 kp 递增应该在每个值上运行至少 30 秒观察，而不是快速递增。

**B**：同意。补充一个安全机制：在 `delta_arm_node.cpp` 中加位置偏差保护——如果任意电机位置偏离目标超过 1.0 rad（约 57°），自动切刹车。

**C**：好建议。加入"偏差过大保护"作为阶段 1 的一部分。

**A**：另外，kd 的经验公式应该调整。对于 GO-M8010-6 通过 RS485 @ 200Hz 控制，kd 太大会导致力矩抖动（因为速度信号噪声大）。建议 kd 上限不超过 kp × 4。

**三人一致赞同完整方案。**

## 最终方案

### 阶段 1：安全恢复（优先级最高）

1. **修复串口热插拔**（见 `serial_hotplug_analysis.md`）
2. **验证/校准 offset**：
   - 在 kp=0.02 下，逐台发送 θ=0° 到各电机
   - 对比三台的物理角度是否一致
   - 若 arm_motor_2 偏离 → 重新测量安装角，更新 `delta_arm.yaml` 中 `motor_offsets_rad[1]`
3. **添加偏差过大保护**：
   ```cpp
   // delta_arm_node.cpp controlLoop() 中
   if (any_online && std::abs(current_joint_angle_ - target_joint_angle_) > 57.0) {
     arm_state_ = ArmState::ERROR;
     sendBrakeCommands();
     RCLCPP_ERROR(..., "位置偏差过大 (%.1f°)，紧急刹车！", ...);
   }
   ```

### 阶段 2：性能优化

4. **逐步提升 kp**：
   | 步骤 | kp | kd | 观察 |
   |------|------|------|------|
   | 2a | 0.10 | 0.10 | 应能缓慢到位（~5s） |
   | 2b | 0.30 | 0.30 | 应快速到位（~1s）|
   | 2c | 0.50 | 0.50 | 接近最终值，观察振荡 |
   | 2d | 微调 | kd=kp×2~4 | 消除振荡/超调 |

5. **更新 `delta_arm.yaml`**：
   ```yaml
   motor:
     kp: 0.30    # 从 0.02 提升到实测最优值
     kd: 0.30    # 同步调整
   ```

### 阶段 3：可选增强

6. 重力补偿前馈：`torque_ff = 0.3`（固定值，或基于角度的 cos(θ) 函数）
7. 速度前馈优化

### offset 正确性判断依据

三台电机同结构安装，合理的 offset 应满足：
- 三者差值不超过 **0.3 rad (17°)**（考虑安装公差）
- 当前 arm_motor_2 的 0.063 与其他两台（0.639, 0.872）差距 0.576~0.809 rad → **明显异常**
- 若实测 arm_motor_2 真实 offset 在 0.5~0.9 范围 → 确认校准有误
